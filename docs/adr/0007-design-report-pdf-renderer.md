# Design Report PDF Renderer

Status: Superseded by [0011 Remove Design Report PDF export and plant display modes](0011-remove-design-report-export-and-display-modes.md).

Historical note: this superseded report-renderer ADR kept its original `0007` number even though an accepted Design Notebook ADR also uses `0007`. Treat ADR 0011 as the current authority for report export behavior.

Canopi Design Reports should be generated by a shared Rust report-rendering service using `printpdf` as the cross-platform PDF writer. The frontend should provide a structured report read model plus a white-background canvas image, and Rust should compose the final paginated PDF bytes. The existing `Platform::export_pdf` path remains a snapshot-style canvas export path and should not become the Design Report renderer.

**Rationale**:
`printpdf` is documented for Linux, macOS, and Windows targets and exposes PDF units, fonts/text, images, page operations, and serialization from one Rust API: https://docs.rs/printpdf/latest/printpdf/. That gives Canopi one renderer implementation for all supported desktop platforms instead of duplicating report layout in Linux Cairo, macOS PDFKit/Core Graphics, and Windows DirectWrite/XPS code. The lower-level `pdf-writer` crate was rejected for this report because its own documentation describes a step-by-step PDF object writer with no validation of required objects, which would push too much PDF machinery into Canopi layout code: https://docs.rs/pdf-writer/latest/pdf_writer/.

**Consequences**:
The Design Report implementation should add `printpdf` to the desktop Rust dependency set and keep the report layout owned by Canopi: page fitting, orientation, text wrapping, table measurement, repeated headers, page breaks, page numbers, legends, and section omission rules are application code, not HTML printing or screenshot reuse. The renderer should embed bundled report fonts rather than depending on system fonts, with coverage for the app's supported Latin, Cyrillic, Japanese, Korean, and Chinese UI languages. This accepts some package-size cost to keep report text deterministic and readable across Linux, macOS, and Windows.

The report renderer should live under a report-specific service boundary, for example `desktop/src/services/design_report/`, and expose a narrow command that receives the current in-memory Design Report input and writes the chosen PDF path. The first canvas page should use the canvas image as a fitted image with a forced white background and orientation chosen from design bounds. Timeline, Budget, Consortium, pinned-name legends, Measurement Guides, and other report sections should be structured PDF content with wrapping and pagination. Empty sections should be omitted before rendering.
