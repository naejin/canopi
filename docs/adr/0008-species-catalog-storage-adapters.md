# Species Catalog Storage Adapters

Canopi will treat the Species Catalog storage engine as an adapter behind the Species Catalog Read Projection interface instead of making one storage engine the product-wide architecture. The desktop app keeps the current Rust and SQLite Species Catalog implementation while the Web Edition v1 uses a browser-native DuckDB-WASM adapter with generated reduced catalog assets; shared UI and workflows should depend on caller-oriented Species Catalog read behavior rather than SQLite, DuckDB, or Tauri-specific details.

**Consequences**:
Web Edition catalog work can use DuckDB-WASM without forcing a risky desktop migration. The Web Edition v1 catalog should be search/filter usable, but must be packaged to fit Cloudflare Pages static-asset constraints: no single catalog, WASM, worker, or data shard may exceed the Pages per-asset limit, and the generated app must keep a conservative file count. This favors a reduced catalog with a strict top-filter set, compressed/sharded data assets, and no full detail payload, hardiness data, height data, or stratum metadata in v1.

Web Edition catalog assets should be generated from the same canopi-data export lineage as the desktop plant DB, not hand-maintained in the website repository. The generator should emit DuckDB-compatible reduced assets, locale/search shards, and image metadata artifacts with deterministic names and size checks so they can move as part of the versioned Web Edition build artifact.

The Web Edition v1 Species Catalog filters are text search by Canonical Name/Common Name, climate zone, habit or growth form, and life cycle. Edibility, hardiness, height, and stratum are not v1 web filters and should not be included in the reduced web catalog. Web v1 Placed Plant presentation should use explicit design colors/symbols and generic fallbacks rather than relying on stratum-backed species metadata.

The Web Edition v1 catalog supports Common Name display and search for the current 11 UI languages: English, French, Spanish, Portuguese, Italian, Chinese, German, Japanese, Korean, Dutch, and Russian. This does not mean shipping the full desktop translation payload or all language columns present in the source database. Language-specific Common Name/search data should be exported as reduced, compressed assets that can be loaded for the active UI language, with Canonical Name and English fallback data available separately where needed.

The Web Edition v1 may include one lazy-loaded hero image per Species when available. The web catalog should ship image metadata only: remote image URL, source, source page or credit link, and any available license/attribution fields. It must not bundle image binaries into the Cloudflare Pages app, prefetch image galleries, proxy images through a backend, or port the desktop native image cache. The current bundled database has first-image-per-species metadata for 74,490 Species, approximately 17 MiB as raw JSON and 4.6 MiB gzipped, so this scope fits the static-asset budget when emitted as a compressed or sharded catalog asset.

The Web Edition v1 Plant Detail view is reduced to the fields already present in the lightweight catalog: hero image, Canonical Name, Common Names, climate zone, habit or growth form, and life cycle. It should omit the desktop detail sections for dimensions, hardiness, height, stratum, uses and edibility, soil, ecology, propagation, risk, related species, and extended taxonomy.

Desktop migration away from SQLite requires measured evidence that the alternative preserves localized Common Name search, filters, detail hydration, and latency while deleting meaningful complexity rather than adding a second engine beside the existing user database.
