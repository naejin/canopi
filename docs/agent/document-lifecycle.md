# Document Lifecycle And Settings

Use this guide when changing `.canopi` load/save, document replacement, dirty state, autosave, settings persistence, or app startup/shutdown.

## Current Boundaries

- `desktop/web/src/app/document-session/actions.ts` exposes user-facing document actions.
- `desktop/web/src/app/document-session/lifecycle.ts` owns canvas runtime attachment, resize observation, autosave timing, settings flush, runtime teardown, and canvas-session publication.
- `desktop/web/src/app/document-session/state-machine.ts` owns explicit desktop Design Session states and intent orchestration: attached/detached readiness, dirty checks, queued loads, autosave execution, teardown snapshots, persistence disposal, and workflow runner disposal.
- `desktop/web/src/app/document-session/replacement.ts` is the platform-neutral authority-handoff seam shared by desktop and Web. Its `attach()` and `replace()` operations own new/loaded normalization, initial load versus explicit replacement, attached/detached store updates, dirty-baseline resets, canvas history/chrome/zoom ordering, and workflow installation. It must import neither Tauri nor browser infrastructure.
- `desktop/web/src/app/document-session/transition.ts` exposes intent-shaped Design Session operations for document actions and lifecycle callers while keeping low-level transition request construction inside the Design Session module.
- `desktop/web/src/app/document-session/store.ts` allocates and owns all Design Session signals and private generations, and is the public state seam for active Design identity, exact non-canvas and detached-canvas revisions, pending loads, autosave failure, and read-only reactive projections. Persistence construction requires its branded factory-created store subtype; structurally compatible stores do not silently acquire private capture authority. Tests that need the production singleton must reset or seed it through the store-owned capability exposed by `desktop/web/src/__tests__/support/design-session-state.ts`; never restore a second raw signal allocator.
- `desktop/web/src/app/document-session/persistence-capability.ts` is the internal capability registry shared only by the store implementation and persistence coordinator. Do not import its raw capture API elsewhere.
- `desktop/web/src/app/document-session/persistence.ts` owns purpose-aware persistence operations. It composes one immutable attached/detached snapshot and keeps session generation, Canvas attachment lifetime, intent ordering, destination-aware write admission, exact acknowledgement, and recovery-result policy behind the operation. Platform adapters couple durable-resource identity to the matching write effect through opaque prepared destinations from `write-admission.ts`; callers never receive raw captured content or publish write success themselves.
- `desktop/web/src/canvas/runtime/lifecycle-owner.ts` retains exclusive runtime release authority across native and Web component unmount. A failed loss-prevention handoff remains retryable there, and the next mount must finish that requested release before constructing another runtime.
- `desktop/web/src/app/document-session/workflow-runner.ts` owns Design Session workflow install/dispose idempotence and retains failed disposer obligations for retry.
- `desktop/web/src/app/document-session/workflows.ts` is the Design Session workflow registry.
- `desktop/web/src/app/contracts/design-ingestion.ts` is the only trust boundary for raw Web Design values. It applies sequential format migrations, generated-schema validation, serde-compatible defaults, typed legacy Object Group resolution, and top-level unknown-field normalization before returning a `CanopiFile`. Browser file, template, and Draft adapters must not cast raw JSON to `CanopiFile`.
- `desktop/web/src/app/contracts/canopi-design-wire.ts` is the Web serialization boundary for canonical `.canopi` JSON. In-memory `CanopiFile.extra` is a normalized compatibility bag; the encoder removes that internal key, restores only unknown entries to the wire root, and keeps every known root field authoritative. Native save/autosave, Saved Object Stamp export, browser download, and current-Design Problem Report attachments must encode through this seam rather than stringify a `CanopiFile` directly.
- `desktop/src/design/format.rs` is the native trust boundary for raw `.canopi` values. It admits the supported version range, applies sequential migrations (including typed legacy Object Group resolution), and only then deserializes the current `CanopiFile` contract. General `CanopiFile` Serde deserialization must not hide compatibility migrations.
- `common-types/canopi-design-conformance.json` is the authored compatibility contract shared by native and Web ingestion. Accepted cases name one normalized document; rejected cases name one generated stable error kind. Both runners execute every case and round-trip every accepted document; the Web runner must traverse the canonical wire encoder between decodes.
- `desktop/web/src/app/consortium/workflow.ts` owns the Consortium sync workflow adapter and effect.
- `desktop/web/src/app/design-edit/` owns non-canvas Design edits through the Design Session store, including committed-versus-visible projection, no-op detection, dirty marking, one active preview transaction, derived reconciliation, and feature-specific writes for Location, Budget Items, Timeline Actions, and Consortiums. Its authority and lifecycle capabilities are private to Design Edit and the store.
- `desktop/src/services/design_files.rs` owns Recent Design recording and listing. Design save/load must not create Design Notebook references; explicit notebook-add behavior belongs to the Design Notebook command/workbench seam. Recent Design listing should check saved path availability before returning entries, prune definitely stale paths from the user DB, and hide ambiguous filesystem states without deleting them.
- The Design Notebook is app-level saved-Design organization. Notebook switching and File-menu Recent Design opens must use `app/document-session/actions.ts` intent-shaped operations such as `openDesignFromPath()` so dirty checks, attached/detached replacement, queued loads, zoom-to-fit, baselines, and workflows stay in the Design Session seam.
- `desktop/web/src/app/document-session/use-canvas-document-session.ts` is a DOM-ref adapter for `CanvasPanel`; keep lifecycle ordering out of the hook.
- `desktop/web/src/app/canvas-runtime/host.ts` publishes `CanvasRuntimeSurfaces` for the live Design Session. Document lifecycle code should consume role-specific canvas surfaces instead of a raw `SceneCanvasRuntime`.
- `desktop/web/src/app/canvas-runtime/app-adapter.ts` is the app-owned bridge for canvas clean state, Design file composition, and settings persistence commands used by runtime core.

## Document Mutation Rules

- No component may replace the active document directly.
- No panel may call document replacement directly.
- Destructive document flows must use the shared guard path: dirty check -> confirm -> replace.
- Design Notebook panels and menu Recent Design rows may initiate switching, but they must delegate to the Design Session action seam and must not call load IPC, state-machine requests, or store replacement directly.
- Production reads of active Design identity and dirty state should use the read-only projections from `app/document-session/store.ts`.
- Production writes to active Design identity, dirty baselines, pending loads, autosave failure, or canvas-clean bridge state must go through `app/document-session/store.ts`.
- Production writes to non-canvas Design fields must go through `app/design-edit/` instead of importing `app/document-session/store.ts` or low-level Design signals directly.
- Previewable non-canvas array edits, such as Timeline Action drags and Consortium drags, must use Design Edit transactions. The UI reads the visible projection, while the authority retains a separate committed Design. A preview replaces one pure whole-Design projector over the latest committed state; preview updates neither persist nor advance the dirty revision, and one or many updates commit at most one dirty revision. Only one long-lived preview is admitted, and a competing begin fails before either projection changes.
- Ordinary Design Edit commands update committed state and reapply the active projector before publishing visible state. Deterministic scene-derived maintenance uses the explicit `reconcileCurrentDesign()` operation, which also replays the projector but does not record independent user intent. Raw committed-edit, reconciliation, and dirty-marking roles are private Design Edit authority capabilities, not methods on the public Design Session store. Do not restore generic caller-controlled `markDirty: false` mutation.
- Save, Save As, recovery, Browser Draft, browser download, handoff, and diagnostic observation capture only committed Design content. Browser autosave observes the committed Design revision, not visible preview identity, so pointer frames do not write Drafts and every committed change remains observable even while the composite dirty flag is already true.
- Replacement guards include the monotonic preview generation. Any later preview activity permanently supersedes that guard. An admitted Design replacement invalidates the predecessor handle before successor publication; late preview, commit, or abort calls are inert and cannot touch or dirty the successor. Design Edit HMR disposal performs the same lifetime rollover for outstanding handles and captures, then permits fresh edits in the new module lifetime.
- Compute an updater or projector and its visible replay before installing committed authority state. Install committed, preview, generation, and terminal outcome state before reactive signal publication so thrown computation publishes nothing partial and reentrant callbacks observe a coherent result.
- Production callers use intent-shaped Design Session operations such as `openDesignSessionFromDialog()`, `openDesignSessionFromPath()`, `openTemplateDesignSession()`, `createNewDesignSession()`, `startAttachedDesignSession()`, `consumeQueuedDocumentLoad()`, `saveCurrentDesign()`, `saveAsCurrentDesign()`, `autosaveDesignSession()`, and `teardownAttachedDesignSession()`.
- Low-level `transitionDocument()` request construction is internal to the Design Session module and state-machine tests. Production callers must not assemble transition sources, dirty guard modes, load callbacks, queue deferral behavior, or attached/detached branch policy.
- Design Session operations decide whether an attached `CanvasDocumentSurface` is available or whether to apply a detached document-state transition.
- Canvas hydration is explicit: only Design Session `attach()` and `replace()` operations may load or replace the runtime document. Never infer canvas replacement from `currentDesign` identity or another reactive non-canvas snapshot update; SceneStore remains authoritative while attached.
- Runtime replacement reserves Settled Scene Authority before it settles/cancels Scene Interaction, detaches inline editors and menu actions, or clears fallible Target presentation. A gesture release hands authority directly to replacement, so no observer can commit into a pre-hydration gap. While that predecessor-to-successor reservation exists, reject reentrant replacement attempts from preparation callbacks; they must never replace the reserved successor. A cleanup failure before the swap removes the reservation and leaves the old Scene and selection mirror authoritative; the Scene coordinator marks that exact outcome with `CanvasDocumentReplacementNotAdmittedError`. Only the coordinator may emit this pre-hydration marker. The document surface restores its prior loaded state, and Design Session discards the token, pending application, and retained authorization before exposing the original reason. Every other hydration-stage failure retains the admitted token and must be replayed. Recovery of an older retained immediate/history operation quarantines the replacement attempt. After a successful swap, the authority invalidates old asynchronous tickets immediately and quarantines the new Scene through history, signal, revision, render publication, and Design-state finalization. It retains its own accepted document snapshot and gives each projection attempt a fresh copy; retry the same replacement if publication fails, reusing its opaque logical replacement token even when normalization creates a fresh object or caller/projection code mutates an object it received. Byte-equivalent content presented with another token is a competing request, not a retry: its event may settle the retained replacement, but remains quarantined and must be attempted again.
- Attached transitions hydrate the canvas session, show chrome, call `zoomToFit()`, and install Design Session workflows. The app-facing `CanvasDocumentSurface.replaceDocument()` requires an opaque replacement token and an explicit idempotent finalizer; do not add optional/no-op defaults that let callers bypass the cross-authority handoff. Design Session retains one application operation across attached or detached retries of a normalized input identity, including its kind, path, and name. That operation owns the accepted file snapshot, Design finalizer, workflow-installation progress, and—when attached—the canvas/token plus replacement, chrome, and zoom progress. A typed busy replacement guard may bypass a new dirty prompt only when an opaque pending-application identity matches both the originally authorized Design baseline and the normalized loaded target key. Preserve a fresh exact Design baseline through the retry load as well; an intervening edit requires a new dirty decision and must never be erased merely because the Scene operation is retained. Distinguish an intervening edit from the same operation's already-applied Design finalizer: the latter intentionally invalidates its input baseline, but its opaque identity and explicit finalization outcome make an idempotent same-target retry safe. A competing target must first finish and quarantine the exact retained operation, then return cancelled/busy so the competing request obtains a fresh dirty decision; never transfer retained authorization to a new file, path, name, kind, or browser Draft identity. A failure after Design or canvas replacement must resume later stages without replacing Scene/Design again; this applies to detached workflow failure as well as attached chrome, zoom, or workflow failure. Settled Scene Authority hydration clears history and invokes the originally accepted finalizer before releasing authority; retry calls resume that retained finalizer and must not register or execute a fresh closure. The returned `CanvasDocumentReplacementReceipt` reports whether this call's closure ran so Design Session can preserve intervening non-canvas edits when only later authority publication needed retry. The finalizer resets both dirty baselines and replaces canonical Design identity as one batched, idempotent stage. It records completion before reactive observers publish; if identity applied and publication then throws, retry must not repeat the stage or erase an intervening non-canvas edit. Do not clear history or reset baselines after release because a release observer may already have admitted and committed a new Scene command.
- Detached transitions update canonical document state, reset dirty baselines, and install Design Session workflows without canvas-only calls.
- Lifecycle callers should use `startAttachedDesignSession()`, `autosaveDesignSession()`, and `teardownAttachedDesignSession()` instead of reassembling Design Session state-machine steps locally. Initial attachment hydrates the canonical Scene synchronously before its returned Promise can yield to a same-turn non-canvas Design edit; do not route mount-existing through an asynchronous acquisition guard. Native surfaces remain unpublished until initial Design hydration succeeds. If that first mount fails, lifecycle uses `abortFailedAttachedDesignSessionStart()` to release the unpublished Canvas lease without a handoff—the canonical store input is still authoritative—then destroys the failed runtime so a later mount can retry cleanly.
- Web file, template, draft, and New Design acquisition stays in `web/browser-design-session.ts`; after resolving an input it delegates replacement and canvas attachment to the shared replacement seam. Decode selected-file and template JSON through `decodeCanopiDesign()` before creating or applying a replacement. Browser app-data hydration decodes each Draft body independently and removes summaries without a valid body, so one corrupt Draft cannot discard unrelated settings, Species data, stamps, or valid Drafts. Issue a monotonic replacement intent and exact guard before awaiting the browser file picker, and let every later New, Template, Draft, or Open intent supersede it so an older picker completion cannot overwrite newer work. If Open finds an exact retained Canvas replacement before the picker, it resumes that operation using its stored Design baseline, returns without opening the picker, and requires the next Open intent to capture a fresh guard. Browser Canvas attachment owns rollback of both its persistence lease and any workflow cleanup obligation retained by a partial installation; retry retained cleanup through the runner instead of abandoning listeners or effects. Browser Draft persistence and desktop native persistence remain separate compile-time adapters.
- Before a Web canvas host is detached or destroyed, `browser-design-session.ts` settles an exact Scene checkpoint into the Design Session store and preserves detached canvas dirty state. Handoff retries when Canvas changes during reactive publication and refuses to publish over a non-canvas Design revision that changed during composition. A later attachment hydrates from that handoff snapshot; never destroy an attached Web runtime while leaving only the older non-canvas snapshot behind.
- Persistence is not a public `serialize` followed by `markSaved` sequence. Manual save, Save As, browser download, browser draft, recovery, teardown handoff, and diagnostic attachment must request the corresponding operation from `app/document-session/persistence.ts`. Writable operations execute an opaque adapter-prepared destination; they own queue admission, one external effect, and exact settlement as a single idempotent operation. Only successful manual/download/draft operations may acknowledge their exact captured baseline; recovery, handoff, and attachment captures never do.
- One persistence service owns one explicit Canvas lease. Design Session or browser lifecycle attachment acquires it; teardown releases it. A second Canvas or duplicate attachment must be rejected until the first is handed off and detached. Attached and detached transition leases are generation checked before every post-await mutation, so an in-flight load cannot hydrate a released runtime and a detached request cannot replace store state behind a live Canvas. Document transitions additionally use issue order through the last pre-application check: only the latest issued transition may begin replacement, including when two loads share the same Canvas attachment epoch. Once replacement applies, a transition issued synchronously by reactive Design publication cannot retroactively cancel its predecessor; the applied predecessor returns `applied` while leaving operation-state publication to the newer transition. Save/recovery callers cannot supply or replace the Canvas. The raw store capture capability is confined to `store.ts`, `persistence-capability.ts`, and `persistence.ts`, not part of `DesignSessionStore`.
- A transition captures an exact replacement guard before its dirty decision and retains that guard through confirmation and asynchronous loading. Any non-canvas revision change, exact Scene-checkpoint change (including byte-equal history divergence), Canvas loaded-state change, attachment change, or session replacement cancels the request before save/discard or replacement can affect the newer state. Generated `updated_at` values are excluded from the guard fingerprint; they are capture metadata, not user edits. If reactive Design publication starts a successor while the predecessor still owns Scene replacement settlement, the typed Canvas-busy capture retains the already captured Design baseline, yields only until the synchronous finalizer releases Scene authority, then resumes Canvas capture. It must not recapture a newer Design baseline after that yield.
- A save completion is current only while its Design Session generation, detached Canvas revision, and Canvas attachment epoch still match. A later replacement or detach/reattach makes the completion inert. Saved-baseline acknowledgement may synchronously trigger reactive reconciliation, replacement, or a nested newer acknowledgement; post-effect divergence repair must revalidate the session, detached Canvas revision, and latest acknowledgement generation before writing. Overlapping manual intents use issue order, not completion order, so an older Save cannot replace the active path selected by a newer Save As. Overlapping recovery attempts likewise publish status only from the latest attempt, and a successful manual save forms a barrier against older recovery results.
- Reserve issue order before capture because Canvas composition can re-enter persistence. Revalidate issue order immediately before the first irreversible acknowledgement and advance the manual-success barrier there, before any fallible reactive clean/path publication can expose an older recovery failure. Once acknowledgement begins, finish that exact operation unless its Design generation or Canvas lease becomes stale; a newer operation issued by reactive clean/path publication remains newer but does not make the partly committed predecessor report stale. Revalidate generation and lease after Canvas acknowledgement, Design baseline publication, and Save As path publication. Operation-state presentation still follows issue order, so an older completion must not hide a newer in-flight save or recovery.
- Design Session operation presentation uses an issue-ordered active-operation registry, separate from transition mutation authority. Show the newest active operation that has reached a presentable phase; when it finishes, restore the newest older operation still in flight. Claim presentation before fallible attachment/capture work, retire every claim on every exit, and retire a predecessor transition only after its successor acquires the Canvas lease and transition intent. A rejected foreign-session operation must reveal the still-active predecessor, while an unresolved superseded transition must never be restored after its successor finishes.
- External writes are admitted FIFO per durable resource and may run concurrently across different resources. Native Design targets use target-family identities, native recovery uses one whole-autosave-store identity because pruning is shared, Browser Draft uses the Draft-partition durable-resource identity `browser-app-data:drafts`, and each browser download uses a fresh non-replacement-blocking identity. Revalidate the captured Design Session generation, Canvas attachment/load epoch, and intent at the head of a delayed lane; ordinary later edits do not cancel an already requested snapshot and remain dirty after exact acknowledgement. Hold the lane through exact settlement so same-resource successors cannot interleave with baseline/path publication.
- A desktop/asynchronous state-machine replacement acquires the persistence write fence after its dirty decision and before loading successor content. The fence drains already admitted replacement-blocking writes, holds through Design/Scene replacement, and delays new predecessor writes. On successful replacement those delayed captures become stale through normal Design/Canvas generation changes; on cancellation or a load/pre-admission failure a still-current delayed write resumes. If Scene admits a replacement and fails before Design finalization, invalidate the predecessor-write epoch before releasing the fence so queued captures cannot write against the partly replaced Scene. Do not acquire the fence before a dirty-confirmation Save, which would make the transition wait on its own queued write. Persistence disposal opens an obsolete gate so queued work can revalidate stale; old fence finalizers must not release or alter a newer gate.
- Browser replacement does not await this asynchronous write fence. Browser Draft storage performs its effect and exact settlement synchronously as one replacement-blocking operation, while a dispatched browser download has a fresh nonblocking identity and may finish after replacement. Browser picker completion remains protected by its exact replacement guard and monotonic intent.
- Keep external file/storage I/O failure separate from exact settlement failure. After a write succeeds, retry the same idempotent settlement through all Canvas, Design-baseline, and Save As path publication phases without repeating I/O or applying write-failure policy. Validate a Save As destination before any acknowledgement. Re-executing one operation with its original prepared destination returns the original execution; it never repeats the effect, and a different destination is rejected after execution begins. Because synchronous Browser Draft storage cannot return an unfinished result, same-stack reentry reports `DesignPersistenceBusyError`; repetition after the first call returns reproduces its recorded result or error.
- An attached Canvas that has not loaded a document still permits a store-authoritative persistence capture. That completion may acknowledge the detached baseline only while the same attachment remains unloaded; hydration that flips loaded state makes the capture stale so it cannot clean later Scene edits.
- Save As changes only the path after the captured snapshot reaches that destination. It must not derive or replace the active Design name from the path. Callers that need the saved content, such as Design Notebook, consume the operation's exact settlement snapshot instead of rereading mutable `currentDesign` after I/O.
- Teardown handoff is loss-prevention, not best-effort logging. Run its fallible authority handoff before clearing autosave, resize observation, queued loads, settings projection, or runtime publication. If an explicit replacement left Scene hydration settling, teardown first resumes that exact opaque operation. Resume normally when its original Design baseline is still current or its finalizer already applied; when a newer non-canvas edit predates finalization, settle the unavoidable Scene swap while preserving current Design authority, then compose that Scene through the normal handoff so the edit remains dirty instead of being erased. If an authoritative Canvas snapshot or workflow disposal still cannot complete, keep its lease, runtime, auxiliary lifecycle, and published surfaces alive, retain failed workflow cleanup obligations, and surface the failure; do not destroy the runtime and expose an older detached snapshot as saveable state. The runtime lifecycle owner retains the failed release after component cleanup and retries it before the next mount. Native asynchronous initialization failure also releases through this owner, destroying the unpublished runtime and clearing auxiliary lifecycle before a later mount retries. Revalidate the Canvas lease after handoff publication so a stale teardown cannot dispose a successor attached by a reactive observer; a stale callback for a no-longer-owned Canvas is inert. Once handoff and workflow disposal succeed, cleanup is terminal: attempt every remaining independent cleanup, unpublish and release ownership even if one fails, and report the aggregate instead of retaining a split, unleased runtime.
- Problem Report attachment capture is a synchronous, non-acknowledging observation routed through the active Design Session persistence service. It must correlate the currently published Canvas identity with that service's existing lease; it does not construct a second save coordinator or claim another lease.
- Browser autosave effects schedule Draft capture in a microtask and coalesce against explicit Draft writes. This keeps capture outside the replacement finalizer while Settled Scene Authority is still admitted and avoids duplicating the explicit post-replacement Draft save. If the intervening write publishes a newer committed Design revision while a task is queued, reschedule that newer identity even when the change is clean reconciliation or the composite dirty state was already true. The queued task must contain capture and settlement exceptions: report autosave failure, keep the Design dirty, and log without creating an unhandled microtask rejection so a later edit can retry. Treat `onDraftSaved` as presentation-only; its exception is logged separately and must not turn a successful Draft acknowledgement into failure.
- Browser Draft identity commits inside the same batched Design-finalization stage as canonical document identity, before reactive publication. A failure before finalization keeps the prior Draft association; a failure in later resumable chrome, zoom, or workflow stages keeps the newly finalized Draft association. This prevents autosave from overwriting either the requested target or predecessor Draft after a partial handoff. Include the Draft identity in the replacement operation key so byte-identical Drafts are still distinct requests.
- Scene live previews are excluded by the Canvas persistence authority, and non-canvas Design Edit previews are excluded by the committed Design capture capability. Persistence may run while either preview is active; it must capture the owning authority's committed state rather than the visible speculative projection.

## Document Authority

- Canvas scene state is owned by `SceneStore`: plants, zones, annotations, groups, Design Object locks, plant species colors, plant species symbols, layers, and canvas session state.
- Non-canvas document state is owned by the document layer: consortiums, timeline, budget, `budget_currency`, location, description, and top-level unknown `extra` fields. Mutations belong behind `app/design-edit/`.
- Non-canvas state must not be pushed into `SceneStore`.
- Canvas state should not be mirrored into standalone signals when a computed value or runtime query surface is enough.
- Design Object lock state is canvas-owned document state. Old files missing per-object `locked` fields load unlocked; new saves must serialize explicit `locked` values.
- New cross-domain sync belongs in a workflow module, not a component effect and not an action-module import cycle.

## Save And Format Contract

- Lightweight web-edition work must still create, open, edit, and export real Canopi Designs rather than a separate web sketch format. Browser storage/file adapters may replace Tauri filesystem dialogs, but they should feed the normal Design Session and save-composition seams. Unsupported web-edition sections should be preserved when loaded where possible and emitted as valid empty sections for new web-created Designs. See `docs/adr/0009-web-edition-uses-design-format.md`.
- The web edition should provide browser-local Design drafts and autosave as the active browser save target, with explicit `.canopi` download/export as the durable portable save path. Browser Drafts are internal autosave/recovery state in v1, not a visible Drafts list, and must not reuse desktop Design Notebook semantics such as Notebook Sections, saved paths, or file reveal actions. Do not make direct save-back to an imported file the v1 foundation, and do not introduce a backend requirement for storing user Designs. Web Edition v1 exports `.canopi` only; do not expose browser PNG, SVG, PDF, CSV, report, or diagnostic bundle export flows. The Web Edition Browser App Shell should present those constraints directly instead of reusing desktop native file chrome. See `docs/adr/0010-web-edition-browser-drafts.md`, `docs/adr/0019-web-edition-canopi-export-only.md`, and `docs/adr/0020-web-edition-browser-app-shell.md`.
- Platform-specific persistence, settings, and file adapters should be selected by the desktop or web Vite entry at compile time, not through runtime web-vs-Tauri feature flags inside shared Design Session modules. See `docs/adr/0021-web-edition-compile-time-adapters.md`.
- Preserve `created_at` from loaded files.
- Preserve loaded document sections on save: timeline, budget, consortiums, description, location, and extra fields.
- Preserve per-object non-visual fields such as plant notes, planted date, quantity, and zone notes.
- Preserve unknown top-level fields through document `extra`.
- Spread `extra` first when composing persisted output so known fields remain authoritative.
- `KNOWN_CANOPI_KEYS` must include `extra`; otherwise `extractDocumentExtra()` can double-nest the `extra` object.
- Top-level annotations belong in the schema. Do not put live annotations under `extra`.
- `serializeScenePersistedState()` emits canvas-owned fields plus placeholders for required non-canvas fields. The app-owned `composeDocumentForSave()` step overwrites placeholders with the Design Session capture's non-canvas values.
- Attached canvas serialization delegates full Design file composition through `app/canvas-runtime/app-adapter.ts`, which calls `composeDocumentForSave()`. Runtime core should not import `app/contracts/document`.
- Attached persistence crosses `CanvasDocumentSurface.captureForPersistence()`, which returns the composed content and an opaque `acknowledgeSaved()` closure for that exact Scene checkpoint. Do not expose separate raw serialization or saved-marker methods again.
- Design file save composition is driven by generated field ownership metadata from `common-types/src/design.rs` via `DOCUMENT_FILE_FIELD_OWNERS`. Do not hand-maintain parallel top-level field merge lists in frontend save code.
- The Web ingestion schema and `CURRENT_CANOPI_FILE_VERSION` are generated into `desktop/web/src/generated/canopi-design-format.ts` from `common-types/src/design.rs`. Generation fails if schema fields diverge from `DESIGN_FILE_FIELDS`, flattened root extras stop being preserved, or Schemars emits a keyword the focused Web decoder does not understand.
- `common-types/canopi-new-design-defaults.json` is the authored authority for static New Design metadata and the standard Layer catalog. Bindings generation validates it and emits typed Rust and TypeScript facts. Desktop and browser composition must inject their own name and timestamp and clone the generated Layer records; Scene defaults consume only the generated Layer catalog, not whole-document defaults. For north-bearing save composition, omitted metadata delegates to the document, explicit `null` remains `null`, and only a genuinely absent legacy document value falls back to zero.
- Shared `extra` subfields need an explicit ownership entry near `composeDocumentForSave()`; currently `extra.guides` is scene-owned.
- Plant Symbols are v3 scene-owned fields: top-level `plant_species_symbols` stores per-species defaults, and each placed plant may carry an optional explicit `symbol`. Old files migrate with an empty `plant_species_symbols` map and missing placed-plant symbols, which render as `round` through the Plant Symbol fallback.
- A species default of `round` is explicit and must be stored as `plant_species_symbols[canonicalName] = "round"`; clearing a species default deletes the key and does not rewrite placed-plant symbols.

## Saved Object Stamp Import And Export

- Saved Object Stamps are personal library entries, not part of normal Design Session save composition, dirty state, autosave, or replacement guards.
- Web Edition v1 keeps Saved Object Stamps browser-local only and does not expose stamp import/export.
- Exporting a Saved Object Stamp writes one valid `.canopi` file containing only the visible canvas objects and captured Object Groups needed for that stamp, with safe default values for required Design fields. Do not export Location, Budget Items, Timeline Actions, Consortiums, description, or non-visual object metadata.
- Importing a Saved Object Stamp from `.canopi` adds a library entry only. It must not open or replace the current Design Session, must not mark the current Design dirty, and must not require the dirty-design guard.
- Stamp import should read `.canopi` files through a lower-level format-loading path or a stamp-specific command that does not record Recent Designs. Do not reuse `design_files::load_design()` for stamp import unless its recent-file side effect is bypassed deliberately.
- Frontend code owns native import/export dialogs; Rust commands should perform file read/write or payload normalization only, matching the existing Linux dialog boundary.

## Export Boundaries

- Structured Design Report PDF export was removed in ADR 0011. Do not reintroduce `app/design-report`, `ipc/design-report`, `commands::design_report`, `services::design_report`, or report-specific PDF dependencies without a new decision record.
- The remaining native PDF path is the canvas snapshot export under `services::export` and `platform::export_pdf`; keep it separate from Design Session save/load and dirty-state handling.
- Web Edition v1 does not expose native-style export flows other than `.canopi` download/export.

## Adding Document Fields

- Add document-level fields to the shared `CanopiFile` contract.

## Dirty State And Autosave

- The composite dirty model has Canvas and non-canvas baselines plus a durable-content divergence latch:
- Canvas dirty state is tracked by exact `SceneHistory` state-identity checkpoints, not history depth. A save acknowledgement can make an older state the baseline while later commands remain dirty; undo becomes clean only when it returns to that exact state, and a divergent branch at the same depth remains dirty.
- Non-canvas dirty state is tracked by `nonCanvasRevision` vs `nonCanvasSavedRevision`. A save acknowledges the captured revision, so edits made during I/O remain dirty. Detached Canvas handoff has its own monotonic revision so an older detached completion cannot acknowledge a newer handoff snapshot.
- Derived reconciliation normally remains clean because it carries no independent user intent. If a successful write captured an older committed Design, `persistenceDiverged` keeps the Design dirty until a current committed snapshot is written; an old write must never leave newer visible content reported clean.
- `committedDesignRevision` is the scheduling identity for Browser Draft autosave. It advances for committed user edits, preview commit, and derived reconciliation, but not for preview updates, preview abort, or save acknowledgement.
- Canvas clean-state updates enter the Design Session through `app/canvas-runtime/app-adapter.ts`. Runtime core should receive a `CanvasRuntimeAppAdapter` clean-state callback instead of importing `app/document-session/store`.
- Do not write `designDirty` directly.
- Recovery autosave captures the same document composition as manual save but never advances a saved baseline. It only publishes current recovery success/failure state.
- Autosave scheduling is owned by `app/document-session/lifecycle.ts`; autosave execution and canvas snapshot composition route through the Design Session state machine; persistence composition remains in `app/document-session/persistence.ts`.
- Canvas session publication updates the surface and ready flag in one signal batch. Teardown may unpublish only when the globally published surface is still the lifecycle's own; a reentrant successor publication must survive stale predecessor cleanup. Fire-and-forget autosave calls must terminate their rejection path through the lifecycle logger.
- Autosave failures surface via `autosaveFailed`.

## Settings Persistence

- The active platform adapter is the durable settings authority: Desktop uses Rust `Settings` in the user DB, while Web Edition uses browser-local app data. Both editions project the same generated `Settings` contract into the same frontend signals.
- Rust `Settings::default()` and the declared Locale, Theme, and Basemap Style catalogs generate `desktop/web/src/generated/settings.ts`; the browser adapter and settings-backed frontend signal initializers consume that runtime authority. Regenerate and check bindings when defaults or those enums change instead of hand-maintaining frontend default tables.
- Keep the shared `Settings` contract limited to preferences with a live runtime or persistence owner. Serde ignores retired unknown keys so older settings JSON remains readable; a retired key must not be emitted by the current serializer or generated TypeScript contract.
- Web Edition settings and personal app data are browser-local convenience state backed by browser storage, not cloud-synced durable state. Settings occupy their own versioned `BrowserAppDataStore` partition. Once that partition exists, Settings writes do not read or serialize Draft bodies; the first transition from the aggregate v1 format or a quota fallback may still touch v1. Each valid v2 record becomes authority for its own resource, while missing or invalid records read through v1 until all four records validate and the immutable v2 authority tombstone commits. Keep web adapters behind the same caller-shaped seams instead of importing browser storage directly from components. See `docs/adr/0014-web-edition-browser-local-app-data.md`.
- Frontend signals are runtime projections.
- `desktop/web/src/app/settings/projection.ts` owns adapter installation, hydration, normalization, snapshotting, mutation, serialized/coalesced persistence, retry state, queued persistence, replacement fencing, and flush behavior. It must not import Tauri IPC or browser storage.
- `platform/desktop.ts` and `platform/browser.ts` install their typed adapter at the compile-time platform composition root. Browser loading remains synchronous so locale, theme, snap, and Plant Spacing hydrate before the first render; Desktop loading may resolve asynchronously.
- Hydration records the normalized snapshot as the durable baseline. Mutations issued during asynchronous bootstrap are replayed over the loaded value and persisted; a load failure establishes the current projection as a retryable fallback instead of permanently disabling persistence. An ordinary unchanged fallback is a no-write baseline so a transient read failure cannot overwrite stored preferences. Handed-off or user intent still requires a save even when its value equals that fallback.
- Persistence advances its durable baseline only after a successful write, never overlaps writes (including synchronous adapter reentry), coalesces newer mutations, and retains a failed latest snapshot for a later mutation or flush to retry. Adapter replacement first admits queued work and fences the successor load behind predecessor settlement so an obsolete save cannot land after the new baseline. If the predecessor write fails, hand its desired snapshot to successor hydration beneath any newer successor mutations and retry it through the successor adapter; never let the successor load overwrite that intent.
- Use `mutateSettingsProjection()` for settings-backed actions.
- Use queued persistence for 60fps-adjacent paths and commit/flush on mouse-up or close as appropriate. Discrete toggles such as snap-to-grid persist immediately in both editions.
- Await `flushSettingsProjection()` when durability gates a lifecycle transition. It joins asynchronous bootstrap without waiting on a retired bootstrap, follows adapter ownership changes, admits the latest queued snapshot, waits for the serialized persistence drain, and rejects on write failure. The Desktop close guard prevents close until that promise settles, coalesces concurrent close requests into one workflow, and keeps the window open on failure.
- Canvas runtime settings-backed commands such as snap-to-grid toggling and Plant Spacing interval commits cross the edition-specific Canvas Runtime App Adapter (`app/canvas-runtime/app-adapter.ts` or `web/browser-canvas-runtime.ts`); runtime core should not import settings projection modules directly.
- Bottom panel tab heights are nullable per-tab settings. `null` means the frontend should use the tab's first-use default; manual resize stores a concrete height for the active tab only.
- Legacy `bottom_panel_height` values are migrated in the Rust settings service: `200` is treated as unset, and any other legacy height seeds Timeline, Budget, and Consortium heights.
- `localStorage` is only a best-effort first-paint cache for theme; a guarded inline script in both HTML entries applies it before module loading, then the active platform settings adapter overwrites it during bootstrap. Cache read/write failure must not prevent parsing, DOM theme application, or settings hydration.
- Theme is light/dark only. Stale `"system"` values are migrated by settings deserialization.

## Tauri Lifecycle Gotchas

- Rust commands return `Result<T, String>` so Tauri serializes errors to the frontend.
- Use types from `common-types` for IPC contracts.
- Map command errors with context, for example `.map_err(|e| format!("Failed to <action>: {e}"))`.
- Use `db::acquire(&db.0, "PlantDb")` for mutex locks; it recovers from poison and warns.
- Use JS dialog APIs from `@tauri-apps/plugin-dialog`; Linux blocking dialogs can deadlock.
- Use `destroy()` instead of `close()` for discard-without-save close paths.
- Events emitted in `setup()` are lost because frontend JS has not loaded yet.
- `tauri.conf.json` uses `beforeDevCommand` with `{ script: "npm run dev", cwd: "web" }` relative to `desktop/`.
- The shell plugin is removed. If external processes are ever needed, use Rust `std::process::Command` from a Tauri command.
